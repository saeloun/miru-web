# Miru 3.0 Launch Narrative

## The Story

Miru 3.0 is the open source billing and time-tracking system for software agencies that want one stable product instead of five stitched-together tools.

This launch is not about matching every feature from every incumbent.

It is about giving a technical services team a sane switching point:

- track time
- review work
- generate invoices
- collect payments
- keep control of the stack

That is the promise.

## Who This Is For

Miru 3.0 is for:

- software agencies
- consulting teams
- product studios
- technical service businesses

Especially teams that are:

- between `5` and `50` people
- already billing for tracked work
- tired of paying for multiple tools that barely fit together
- technical enough to value open source, CLI, API, and self-hosting

## What They Use Today

Most of these teams are on a messy stack.

Something like:

- one product for time tracking
- another for invoicing
- Stripe for payments
- spreadsheets or Notion for cleanup
- Jira, Slack, and git as the actual work trail

The result is predictable:

- time leaks
- invoices take too long
- reporting is messy
- nobody fully trusts the system

## Why They Switch

They do not switch because Miru has more checkboxes.

They switch because Miru aims to give them:

- one place for time, invoices, payments, and team operations
- open source code they can inspect and run
- self-hosting when they want control
- a CLI and API that fit technical workflows
- a product stable enough to move real work onto, once the launch checklist is green

That last point matters most.

If the product is not trusted, nothing else matters.

## What Miru Replaces

Miru 3.0 is trying to replace the billing operations stack for a technical services team.

That means replacing:

- fragmented time tracking
- manual invoice cleanup
- payment visibility spread across tools
- disconnected team and client operations

It does not need to replace every adjacent tool on day one.

It needs to replace the painful core.

## What Is Stable Enough Today

Miru 3.0 should be presented as ready only when these are true on production:

- sign in and invite flows work
- time tracking is reliable
- historical entries still load
- PTO and holiday entries show correctly
- invoices load, sort, search, and scroll correctly
- payments are visible and usable
- dashboard does not break on real production data
- CLI workflows work for real user accounts
- deploy and restore are understood
- backups have been verified recently
- monitoring, alerting, and incident ownership are in place for core routes
- email delivery and payment-webhook paths are behaving reliably on the launch path

That is enough for launch.

Not tiny. Not fake. Enough.

## What We Are Not Claiming

Miru 3.0 is not claiming:

- full category parity with every incumbent
- perfect mobile parity
- every post-launch workflow already finished
- every future AI or agent workflow included at launch

That is fine.

The launch should be honest:

- stable core workflows now
- stronger technical control now
- deeper expansion after trust is earned

## The Migration Promise

The launch promise should include a simple migration story:

`Move your time tracking and invoicing workflow to Miru in a weekend, if your data is clean and the supported migration path fits your stack.`

That only works if we keep the migration path real:

- importers for the chosen switching path
- clear docs
- readable data model
- exports that keep customers from feeling trapped

If migration is weak, the launch story is weak.

If migration support is slow, importer behavior is unreliable, or production-sized datasets have not been exercised, do not make the weekend-switch promise in public launch copy.

## Positioning

The sentence to keep repeating:

`Miru 3.0 is the open source billing and time-tracking system for software agencies that want one stable product instead of five stitched-together tools.`

The supporting points:

- open source
- self-hostable
- built for technical teams
- API and CLI included
- time-to-invoice is shorter
- one stable operating surface instead of a patchwork, once the release criteria are actually met

## What Happens After Launch

After the launch is trusted, then expand into:

- agent-ledger billing
- mobile wedge
- deeper migration tooling
- richer reports
- more workflow automation

That order matters.

The trap is launching a bigger product.

The win is launching a trusted one.

Immediately after launch, the first hardening loop should focus on:

- support response time
- restore confidence
- performance on real production accounts
- importer cleanup on real customer datasets
- bug-response speed on core flows

## Use This In Launch Copy

### Short Version

Miru 3.0 is the open source billing and time-tracking system for software agencies that want one stable product instead of five stitched-together tools.

### Medium Version

Miru 3.0 gives software agencies one place to track time, review work, generate invoices, collect payments, and run billing operations without depending on a pile of disconnected SaaS tools. It is open source, self-hostable, includes a real CLI and API, and is built for teams that care about control and reliability. This launch claim is only valid when the production launch checklist is green.

### Launch Post Framing

This release is about switchability and trust.

Not feature bingo.

Not roadmap theater.

A real agency should be able to move the core billing workflow to Miru 3.0 without feeling reckless, if the supported migration path matches the customer and the production launch checklist is green.

That is the standard.
